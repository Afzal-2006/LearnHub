1. Big O Notation and Algorithmic Efficiency

Data Structures and Algorithms are about solving problems efficiently. A working solution is not always a good solution. If an algorithm takes too long on large input sizes, it becomes impractical in real systems.

Big O notation helps measure how time or space requirements grow as input size increases. It does not focus on tiny constant details. Instead, it focuses on growth patterns as the amount of data becomes large.

Why Efficiency Matters

A simple loop over 10 items feels instant. A nested loop over a million items can become extremely slow. The ability to estimate performance before running code is one of the most important programming skills.

Common Time Complexities

Complexity Meaning Typical Example
O(1) Constant time Array index access
O(log n) Logarithmic time Binary Search
O(n) Linear time Single loop
O(n log n) Efficient sorting class Merge Sort, Quick Sort average case
O(n²) Quadratic time Nested loops, Bubble Sort

Examples

// O(1)
console.log(arr[3]);

// O(n)
for (let i = 0; i < n; i++) {
  console.log(i);
}

// O(n²)
for (let i = 0; i < n; i++) {
  for (let j = 0; j < n; j++) {
    console.log(i, j);
  }
}
💡 Big Idea: Big O is not about writing scary math. It is about understanding how solutions scale when the input becomes large.

2. Arrays and Linked Lists

Arrays and linked lists are both linear data structures, but they work very differently. Understanding their trade-offs helps you choose the right one for a problem.

Arrays

Arrays store values in contiguous memory. This makes indexed access extremely fast, usually O(1).

const numbers = [10, 20, 30, 40];
console.log(numbers[2]); // 30

However, inserting or deleting at the beginning or middle can be costly because other elements may need to shift.

Linked Lists

Linked lists are made of nodes. Each node stores data and a reference to the next node.

class Node {
  constructor(data) {
    this.data = data;
    this.next = null;
  }
}

Linked lists are more flexible for insertion and deletion when you already have a reference to the correct location. But they do not support fast indexed access like arrays do.

Comparison

Operation Array Linked List
Indexed Access Fast Slow
Insert at Front Slow Fast
Memory Layout Contiguous Distributed nodes
Remember: Arrays are excellent for direct access. Linked lists are useful when structural changes happen often.

3. Stacks and Queues

Stacks and queues are simple but powerful structures that control the order in which items are processed.

Stacks

A stack follows LIFO — Last In, First Out. The last item added is the first one removed.

  • Push = add item
  • Pop = remove top item
  • Peek/Top = inspect top item

Stacks are used in undo systems, browser history, recursion, and expression evaluation.

Queues

A queue follows FIFO — First In, First Out. The first item added is the first one removed.

  • Enqueue = add item
  • Dequeue = remove front item
  • Front = inspect first item

Queues are useful in task scheduling, printer systems, buffering, and breadth-first search.

Structure Rule Common Use
Stack LIFO Undo, recursion, backtracking
Queue FIFO Scheduling, BFS, buffering
💡 Tip: Even simple structures become extremely powerful when combined with the right problem-solving technique.

4. Trees, Binary Search Trees, and Hashing

Trees store hierarchical data. A tree has nodes connected in parent-child relationships. They are used in file systems, parsers, search structures, and database indexing.

Binary Trees

In a binary tree, each node can have at most two children: left and right.

Binary Search Tree (BST)

In a BST:

  • Left child values are smaller than the parent
  • Right child values are larger than the parent

This structure can make searching efficient if the tree remains balanced.

Hash Tables

A hash table stores key-value pairs and uses a hash function to decide where a value should be placed. It is one of the most practical structures in real software because average lookup can be very fast.

Structure Best Strength Typical Complexity
BST Ordered searching O(log n) if balanced
Hash Table Fast average lookup O(1) average
Important: Trees are good when order matters. Hash tables are great when fast direct lookup matters more than sorted order.

5. Sorting and Searching

Sorting arranges data in order, which often makes other operations easier and faster. Searching finds a target value within data.

Bubble Sort

Bubble Sort repeatedly compares adjacent elements and swaps them if they are in the wrong order. It is easy to understand but inefficient for large input sizes.

for (let i = 0; i < arr.length; i++) {
  for (let j = 0; j < arr.length - i - 1; j++) {
    if (arr[j] > arr[j + 1]) {
      [arr[j], arr[j + 1]] = [arr[j + 1], arr[j]];
    }
  }
}

Binary Search

Binary Search works only on sorted arrays. It repeatedly cuts the search space in half, making it much faster than linear search on large sorted data.

function binarySearch(arr, target) {
  let left = 0, right = arr.length - 1;

  while (left <= right) {
    let mid = Math.floor((left + right) / 2);

    if (arr[mid] === target) return mid;
    if (arr[mid] < target) left = mid + 1;
    else right = mid - 1;
  }

  return -1;
}

Sorting Comparison

Algorithm Typical Complexity Note
Bubble Sort O(n²) Simple but slow
Merge Sort O(n log n) Stable and efficient
Quick Sort O(n log n) average Very practical and fast in average cases

6. DSA Best Practices and Final Quiz

Strong DSA thinking is not about memorizing every algorithm. It is about recognizing patterns, choosing suitable structures, and understanding performance trade-offs.

Best Practices

  • Always think about input size before choosing an approach.
  • Pick a data structure that matches the required operations.
  • Prefer clarity first, then optimize where necessary.
  • Understand why an algorithm works, not just how to type it.
  • Practice problem-solving consistently to build intuition.
